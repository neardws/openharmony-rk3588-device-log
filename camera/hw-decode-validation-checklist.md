# 摄像头 RTSP 硬件解码上板验证 Checklist

> 目标：验证 OpenHarmony 镜像在板端接入 IPC 摄像头 RTSP 视频流后，是否真正走了硬件解码路径，而不是 CPU 软解 fallback。

## 0. 验证对象

- 摄像头：宇视 IPC
- 输入协议：RTSP
- 主码流：H.264
- 推荐先测 URL：

```text
rtsp://admin:a1234567~@192.168.1.13:554/media/video1
```

## 1. 验证前提

以下条件建议先确认：

- [ ] 板子已刷入新的 OpenHarmony 镜像
- [ ] 系统可正常启动
- [ ] `eth1` 可与 IPC 摄像头直连
- [ ] 摄像头可 ping 通
- [ ] HDC / shell 可进入系统
- [ ] `/vendor/lib/passthrough/indirect/` 下可见硬解相关库
- [ ] `system/bin/hcodec_demo` 存在
- [ ] `system/etc/codec/codec_caps.xml` 存在

建议先核对这些库：

- [ ] `librockchip_mpp.z.so`
- [ ] `librockchip_vpu.z.so`
- [ ] `libomxvpu_dec.z.so`
- [ ] `libomxvpu_enc.z.so`
- [ ] `libOMX_Pluginhw.z.so`
- [ ] `librga.z.so`

## 2. 网络与输入源验证

### 2.1 配置网口

```bash
ifconfig eth1 192.168.1.100 netmask 255.255.255.0 up
```

- [ ] `ifconfig eth1` 显示地址正常

### 2.2 Ping 摄像头

```bash
ping -c 3 192.168.1.13
```

- [ ] 可达
- [ ] 丢包率低
- [ ] 时延稳定

### 2.3 记录当前码流配置

建议先通过已有 `camera/api.md` 方法确认：

- [ ] 主码流分辨率
- [ ] 帧率
- [ ] 码率
- [ ] 编码格式确认为 H.264

## 3. 先保留一份软解基线数据

在做硬解前，建议先保留一组软解对照组。

建议记录：

- [ ] CPU 占用
- [ ] 3 秒短跑吞吐
- [ ] 30 秒持续解码稳定性
- [ ] 是否掉帧
- [ ] 是否 OOM / 卡死

推荐至少保留：

- [ ] 子码流软解基线
- [ ] 主码流软解基线

## 4. 硬解验证目标

首轮验证目标不要太大，建议按这个顺序：

### 第一阶段

- [ ] RTSP 输入可进入硬件 decoder
- [ ] 能稳定输出 YUV frame 或 null sink
- [ ] CPU 明显低于软解

### 第二阶段

- [ ] 持续跑 30 秒以上稳定
- [ ] 无明显掉帧
- [ ] 无媒体服务崩溃

### 第三阶段

- [ ] 导出单帧 / 少量帧
- [ ] 验证图像内容正确

## 5. 需要记录的关键日志

验证时建议同时抓以下日志：

- [ ] `media_service`
- [ ] `av_codec_service`
- [ ] `codec`
- [ ] `omx`
- [ ] `mpp`
- [ ] `rockchip`

可记录为：

- 时间戳
- 启动命令
- 日志起止区间
- CPU 占用截图 / 文本
- 帧率与 speed 输出

## 6. 如何判断“真的走了硬解”

至少满足下面 3 条中的 2 条，才建议判定为“硬解成立”：

- [ ] CPU 占用明显低于软解
- [ ] 日志中出现硬件 codec / OMX / MPP 相关调用痕迹
- [ ] 高分辨率主码流下仍能实时稳定处理

如果只是“能出图”，但：

- CPU 和软解差不多
- 没有任何硬件路径日志
- 高码率一上来就掉速

那就要怀疑仍是软解 fallback。

## 7. 建议的测试顺序

### 7.1 先测子码流

- [ ] 分辨率低
- [ ] 先确认硬解路径是否基本可用

### 7.2 再测主码流

- [ ] 提高解码压力
- [ ] 观察实时性和 CPU 变化

### 7.3 再做长时间稳定性验证

- [ ] 持续 5 分钟
- [ ] 持续 10 分钟

## 8. 验证输出建议

首轮验证建议优先使用：

- [ ] null sink
- [ ] YUV frame 输出

不建议第一轮就追求：

- [ ] GUI 播放窗口
- [ ] 完整播放器
- [ ] 边显示边推理

原因：这些会引入额外变量，不利于判断问题到底出在：

- 拉流
- 解码
- 颜色转换
- 显示链
- 推理链

## 9. 失败时优先排查什么

### 症状 A：RTSP 不通

优先查：

- [ ] `eth1` IP 是否正确
- [ ] 摄像头供电 / 网线
- [ ] 摄像头 RTSP 账号密码

### 症状 B：能拉流但解码失败

优先查：

- [ ] 输入是否是标准 H.264 码流
- [ ] 码流 profile / level 是否超出 decoder 支持
- [ ] 解码器初始化失败日志

### 症状 C：能出帧但 CPU 很高

优先查：

- [ ] 是否走了软解 fallback
- [ ] 只是 demux 成功，decoder 仍为软件实现
- [ ] 输出格式转换是否吃掉大量 CPU

### 症状 D：前几秒正常，长跑崩溃

优先查：

- [ ] buffer 泄漏
- [ ] 媒体服务重启
- [ ] frame queue 堵塞
- [ ] 大分辨率下内存压力

## 10. 首轮验证通过标准

建议把“首轮通过”定义为：

- [ ] 主码流 RTSP 输入可稳定处理至少 30 秒
- [ ] CPU 相对软解有明确下降
- [ ] 日志能证明硬件解码路径被调用
- [ ] 无明显掉帧 / 卡死 / 花屏

## 11. 通过后下一步

如果首轮通过，后续按下面顺序推进：

- [ ] 导出单帧 YUV / JPEG
- [ ] 接 RGA 转 RGB
- [ ] 接 NPU 前处理
- [ ] 再考虑 GUI 显示或完整业务链
