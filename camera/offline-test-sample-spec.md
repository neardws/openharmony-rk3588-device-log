# 离线硬解测试样本规范

> 目标：在暂时没有板子、没有摄像头的情况下，先准备一组标准化的离线码流样本，未来可直接用于 `hcodec_demo`、MPP 或其他硬解路径验证。

## 1. 为什么要先准备离线样本

离线样本的价值在于：

- 不依赖 RTSP 网络环境
- 不依赖 IPC 摄像头在线状态
- 不依赖真实板端接线
- 可以先专注验证“解码器本身是否工作”

推荐把验证拆成两个阶段：

### 阶段 A

```text
离线样本 -> hcodec_demo / 硬解器
```

### 阶段 B

```text
RTSP 摄像头 -> 裸流提取 -> hcodec_demo / 硬解器
```

这样做的好处是，板子一到就能先跑阶段 A，快速定位问题是在：

- 样本
- 硬解器
- 还是 RTSP 输入层

## 2. 推荐样本类型

建议至少准备下面几类：

### 2.1 H.264 Annex-B 裸流

文件示例：

- `test-h264-720p-30fps-10s.h264`
- `test-h264-1080p-30fps-10s.h264`
- `test-h264-1520p-30fps-10s.h264`

### 2.2 H.265 Annex-B 裸流

文件示例：

- `test-h265-1080p-30fps-10s.h265`
- `test-h265-4k-30fps-10s.h265`

### 2.3 可选压力样本

用于后续稳定性测试：

- `test-h264-1080p-60fps-30s.h264`
- `test-h265-4k-60fps-30s.h265`

## 3. 强烈建议的输入格式

对 `hcodec_demo` 来说，最稳的输入格式是：

- elementary stream
- Annex-B
- 带 start code

也就是：

- H.264：`.h264`
- H.265：`.h265` / `.hevc`

不建议把第一批样本做成：

- `mp4`
- `mkv`
- `ts`
- `flv`

因为那样会把“容器 demux”问题混进来，降低定位效率。

## 4. 推荐样本元数据

每个样本建议配一份简单说明，至少包含：

- 编码：H.264 / H.265
- 分辨率：如 1280x720 / 1920x1080 / 2688x1520 / 3840x2160
- 帧率：25 / 30 / 60 fps
- 时长：10s / 30s
- Profile：Baseline / Main / High / Main10
- GOP 信息：I 帧间隔大概多少
- 是否带 B 帧
- 是否带 start code
- 文件大小
- 生成方式或来源

建议可以配一个 `samples.md` 或同名 `.json` 元数据文件。

## 5. 推荐的第一批最小样本集

如果只做最小可用集合，建议先准备这 4 个：

### 最小集合 A

- `test-h264-720p-30fps-10s.h264`
- `test-h264-1080p-30fps-10s.h264`
- `test-h265-1080p-30fps-10s.h265`
- `test-h264-2688x1520-30fps-10s.h264`

这四个样本的作用分别是：

- 720p：最小门槛验证
- 1080p H.264：主流硬解验证
- 1080p H.265：验证 HEVC 路径
- 2688x1520 H.264：贴近当前 IPC 摄像头主码流

## 6. 与当前项目的对应关系

当前项目中，IPC 摄像头主码流大致对应：

- H.264
- 2688 × 1520
- 30fps（API 文档实测配置）

因此未来最关键的离线样本之一应当是：

- `test-h264-2688x1520-30fps-10s.h264`

这样即使没有摄像头，也能尽量贴近真实上板目标。

## 7. 样本来源建议

样本来源优先级建议如下：

### 优先级 1：自生成可控样本

优点：

- 参数完全可控
- 容易记录 profile / fps / 分辨率
- 可复现

### 优先级 2：从已有 RTSP 流转储

优点：

- 最贴近真实摄像头

缺点：

- 当前没摄像头时无法立即准备

### 优先级 3：公开测试样本二次转换

优点：

- 获取快

缺点：

- 元信息可能不够规整
- profile / GOP 不一定理想

## 8. 未来板端验证如何使用这些样本

板子到位后，建议按顺序测试：

1. `720p H.264`
2. `1080p H.264`
3. `1080p H.265`
4. `2688x1520 H.264`
5. 长时间压力样本

这样可以逐步确认：

- 功能是否可用
- H.264 / H.265 是否都支持
- 高分辨率下是否稳定

## 9. 样本文件命名规范建议

建议统一命名格式：

```text
test-<codec>-<resolution>-<fps>-<duration>.<ext>
```

例如：

- `test-h264-720p-30fps-10s.h264`
- `test-h264-1920x1080-30fps-10s.h264`
- `test-h265-3840x2160-30fps-10s.h265`
- `test-h264-2688x1520-30fps-10s.h264`

## 10. 当前建议

当前没有板子和摄像头时，最有价值的是先把样本规范定下来，而不是急着找随机视频文件。

建议后续把样本目录单独放成：

- `camera/samples/README.md` 或
- `camera/test-samples/README.md`

并在里面维护：

- 文件名
- 参数说明
- 来源
- 推荐用途

## 11. 一句话总结

> 离线样本的目标不是替代摄像头，而是让“硬解器本身验证”脱离真实硬件依赖，先跑起来。
